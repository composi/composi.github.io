Effects

Launching Effects from an Action

In @composi/core runtime an effect is a function that sends a message. Effects are asynchronous. Effects let you interact with the world outside the runtime. An effect can create a timer, fetch data, persist data, log data, or open a browser alert. The section covering init explained how to launch an effect at startup by providing init with a second value in its array. Similarly, the section on update showed that an action always returns state surrounded by braces. That's so you can include an optional second value as an effect.

Separate but Equal

Separating effects out into their own actions makes it easier to reason about the actions that deal with state management.

Below is an example of a runtime program that has two effects but does not return a view. Instead the view returns undefined. This program's init has state and an effect. The effect causes a counter to start at load time. When it reaches five, it triggers another affect to alert the final result. If you open the web inspector console, you'll see the count progress, followed by an alert upon completion.

See the Pen @composi/core + runtime Effects-1 by Robert Biggs (@rbiggs) on CodePen.

When your program is very simple it may not seem like such a big deal to separate effects from the other logic. But as a program becomes more complex, isolating effects from business logic results in code that is easier to reason about.

batchEffects

Sometimes you have a lot of effects that you need to run together. Or you have one big effect doing alot of things that you wish you could break down into smaller pieces. @composi/core provides batchEffect for those purposes. It allows you to create a map of effects that you can execute one after the other. Be aware that effects run asynchronously and independently of each other. So, event though they are being executed in order, they will probably terminate at different times. If you want to execute effects sequentially, one after the other, then you should look into running them as ES6 async/await functions.

mapEffects

Another useful function for effects is mapEffects. You would use this the same way you use tagged unions to enable views to use actions. With mapped effects you can access from one program the effect of another program. This opens up a channel of communication between a parent program and a child program.

To create mapped effects you first need to import it into your project.

import { h, render, run, mapEffects } from '@composi/core'

const counter = {
  init() {
    return [0]
  }, // State is an integer to count
  update(state, message) {
    return [state + 1] // Increment the state
  },
  view(state, send) {
    // add later...
  }
}
// Effect we will use with counter:
const counterMessage = message => ({
  type: 'counterMessage',
  data: message
})

// Capture the state and effect of counter program to be used by its parent:
const [counterState, counterEffect] = counter.init()

// Create a new init function that we will use with counter.
// Here we map the effects:
const init = () => [{counterState}, mapEffect(counterEffect, counterMessage)]

In the following Codepen example, the child program is being rendered by the parent program. But the child has its own local state. To enable the parent to know that state and use it to render the child, we use a mapped effect. The mapped effect allows the parent to know what the child's state is. It has access to the state to print it out in its own scope, but the state is owned by the child.
